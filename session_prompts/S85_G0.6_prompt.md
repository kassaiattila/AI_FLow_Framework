## FELADATOK

### LEPES 1 — Webhook router architectural decision

**STOP candidate** — decide in-session, escalate to architect subagent if no consensus:

- **Path A (sync sink)**: router calls `await IntakePackageSink(repo).handle(pkg)` after `adapter.enqueue()`. Status code becomes **201 Created** (semantic: persisted). Update 8 existing webhook tests + the `WebhookAcceptedResponse` model. Adapter's in-memory queue becomes vestigial (drained immediately) — could remove or keep for future async migration.

- **Path B (async worker)**: keep router at 202. Add `aiflow.sources.worker.WebhookDrainWorker` that loops `await process_next(adapter, sink)` with backoff. Wire into FastAPI lifespan startup. Keeps webhook fast (HTTP returns before DB write) but introduces durability gap (in-memory queue → process crash drops accepted packages).

**Default recommendation: Path A.** Webhook scale is low (few-per-second), DB write is sub-100ms, and Path B's durability gap is unacceptable without a real persistent queue (Redis/SQS). Path B may revisit when the volume requires it.
