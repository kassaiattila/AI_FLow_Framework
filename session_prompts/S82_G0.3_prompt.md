## KONTEXTUS

G0.2 locked the persistence pattern: `IntakePackageSink` orchestrates
`resolve_mode_and_associations() → repo.insert_package() → source.package_persisted`
and adapters stay unchanged. Caller composes them via `process_next(adapter, sink)`.

G0.3 applies the pattern to two more adapters — **File** and **Folder** —
which share `aiflow/sources/_fs.py` for filesystem materialisation, so one
session covers both cleanly.

**STOP FELTETEL check before coding:** `FileSourceAdapter` produces
`FILE_UPLOAD` rows, and `api/v1/intake.py:upload_package` already writes
`FILE_UPLOAD` too. If the sink-wired FileSourceAdapter is driven from the
same HTTP entry point, that's a double-writer. Decide in-session:
- **Path A (preferred):** refactor `upload_package` to build an
  `IntakePackage` and delegate to `sink.handle()`, collapsing the two
  persist paths into one. `FileSourceAdapter` remains for worker-driven
  internal uploads.
- **Path B:** keep the HTTP route as-is, wire `FileSourceAdapter` only for
  non-HTTP callers (no caller today; document the intended consumer).
