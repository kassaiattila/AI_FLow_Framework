## FELADATOK

### LEPES 1: Architect-review the persistence pattern

Before touching code, draft a 1-page ADR note (inline in the forthcoming PR body, no separate file) weighing:

- **Option A — `SourceAdapter.persist(repo)` method.** Keeps lifecycle cohesive; every adapter subclass implements persistence. Cost: couples adapters to `IntakeRepository`, making unit-testing require a repo double.
- **Option B — external orchestrator/sink.** A new thin class (e.g. `IntakePackageSink`) sits between `fetch_next()` and `acknowledge()`, calls associator helper + `insert_package()`. Adapters remain testable in isolation. Cost: one extra indirection layer, behaviour split across two modules.

Call the `architect` agent with both options and the audit table from S80. Pick one and record the verdict. Default recommendation (no architect input yet): **Option B (orchestrator/sink)** — adapters stay focused on upstream I/O, persistence stays a single testable path.
