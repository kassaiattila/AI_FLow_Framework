## FELADATOK

### LEPES 1: CHECK constraint design decision

PostgreSQL row-level CHECK cannot reference another table directly, so `description_count = 0 OR association_mode IS NOT NULL` needs one of:

**Option A (preferred) — BEFORE INSERT/UPDATE trigger on `intake_packages`:**
- Trigger body `SELECT COUNT(*) FROM intake_descriptions WHERE package_id = NEW.package_id;`, raises exception if count > 0 AND `NEW.association_mode IS NULL`.
- Enforced at write time; idempotent; downgrade drops the trigger.
- Disadvantage: the description rows must exist BEFORE the package UPDATE that sets NULL (insert order matters — packages are normally inserted first, then descriptions). Need to decide whether the trigger fires on INSERT of the package (where description_count is still 0) or on UPDATE (where it can be enforced).
- Decision: fire on BOTH INSERT and UPDATE. Since descriptions always have FK to package_id, they cannot exist before the package row — so INSERT-time check is moot (count is 0, passes). UPDATE-time check catches the case of a package transitioning mode to NULL after descriptions were added.
- But the real concern is: a package is INSERTED with descriptions already attached (one transaction). Trigger on `intake_descriptions` AFTER INSERT checking parent's mode is cleaner.

**Option B — AFTER INSERT trigger on `intake_descriptions`:**
- After any description is inserted, assert the parent package has a non-NULL `association_mode`. If not, raise.
- Cleaner semantics: "no description may exist under a parent with NULL mode".
- Downside: deferred error (only trips when a description is added).

**Option C — GENERATED column `has_descriptions` + plain CHECK:**
- `has_descriptions BOOLEAN GENERATED ALWAYS AS (...)` — not possible from another table in PG generated columns.
- NOT VIABLE.

**Recommendation:** **Option B.** Simpler trigger, matches the semantic invariant. Pseudo:
```sql
CREATE FUNCTION intake_require_association_mode() RETURNS TRIGGER AS $$
BEGIN
  IF (SELECT association_mode FROM intake_packages WHERE package_id = NEW.package_id) IS NULL THEN
    RAISE EXCEPTION 'intake_descriptions inserted but parent package.association_mode IS NULL (package_id=%)', NEW.package_id;
  END IF;
  RETURN NEW;
END $$ LANGUAGE plpgsql;

CREATE CONSTRAINT TRIGGER intake_descriptions_require_mode
  AFTER INSERT ON intake_descriptions
  DEFERRABLE INITIALLY IMMEDIATE
  FOR EACH ROW EXECUTE FUNCTION intake_require_association_mode();
```

**DECISION POINT:** if this approach is wrong (e.g. the architecture council wants a pure CHECK via a materialized `has_descriptions` boolean on `intake_packages`), surface it as STOP FELTETEL before writing the migration.
