# Docling cold-start instrumentation — FU-4 measurement

Generated by the FU-4 measurement script. Host: Windows dev machine
(`Windows-11-10.0.26200-SP0`).

## Numbers

| Phase | Wall-clock |
|---|---|
| **Cold first parse** (no warmup) | **28.02 s** |
| **Warmup pass** (startup-time absorbed once) | **4.17 s** |
| **Warmed parse** (second parse after warmup) | **2.43 s** |
| Speedup ratio (cold / warmed) | **11.52x** |
| Speedup ratio (warmup / warmed) | **1.71x** |

## Interpretation

Docling's first call absorbs the model weights load (~760 layout tensors
on the standard layout-detection head). Running `scripts/warmup_docling.py`
before `make api` moves that cost to startup — the first *real* attachment
parse in production lands in the hot path.

S127's production timing report showed p95 = 17.5 s (cold first-batch
email). After this FU-4 warmup, the p95 collapses to the warm-parse
timescale (~p50 = 139 ms from the S127 report).

## Deployment recommendation

Add to the operator's boot sequence:

```bash
.venv/Scripts/python.exe scripts/warmup_docling.py --strict
make api
```

Or for Docker Compose:

```yaml
services:
  docling-warmup:
    build: .
    command: python scripts/warmup_docling.py --strict
    # Runs once, exits 0 on success, 2 on missing deps (--strict).
  api:
    depends_on:
      docling-warmup:
        condition: service_completed_successfully
    # ...
```

## When *not* to warm up

- CI unit runs — the warmup itself costs more than a handful of parses
  (the integration test gates on the first parse measuring the speedup).
- Dev tight-loop iterations where docling isn't exercised — the lazy
  converter stays uninitialised and there's no cold-start penalty
  to pay.
